Case Study 01 - Benefit Overview Redesign
Cigna / Evernorth · Product Design · 2022

The Problem

Dense prose. Zero hierarchy.

Members navigating health decisions under stress were confronted with long paragraphs of benefit descriptions, no visual hierarchy, no member-specific data, no clear action paths. Confusion drove support calls. Ambiguity eroded trust.

Core Issues

Unstructured text with no scan path. No personalization of member data. Multiple WCAG failures. Unclear next actions for checking coverage, comparing plans or finding a provider.

Design Constraints

Healthcare compliance requirements. Real-time member data integration. Legacy CMS. Cross-functional alignment across product, clinical, legal and engineering.

Benefit Overview · Modular card system · Accessibility · Cigna / Evernorth
Strategy

Modular by design.

Each benefit category became a self-contained card surfacing real member data, deductibles, coverage tiers, in-network status in context. The member no longer needed to read everything to find what was relevant.

Left: Benefit card component · Right: Mobile view
Process

How the work happened.

01
Research & Discovery

Conducted internal stakeholder discussions, analyzed previously collected member feedback and facilitated Zoom empathy sessions with 6 users. Identified key pain points around dense healthcare content, poor accessibility and cognitive overload.

02
Information Architecture & Ideation

Audited against WCAG 2.2 and simplified 40+ disconnected content areas into 8 clear card-based benefit categories. Created wireframes focused on scannability, faster decision-making and task-oriented navigation.

03
Design & Prototyping

Designed accessible, reusable UI components with strong visual hierarchy and token-driven consistency. Built interactive prototypes validated through user testing to ensure members could quickly understand coverage details, costs and recommended actions.

04
Accessibility Testing & Handoff

Completed VoiceOver, NVDA, keyboard navigation, focus management and color contrast validation before dev handoff. Delivered annotated Figma specs, accessibility guidance and component documentation for seamless engineering implementation.

The Problem

Cluttered. Inaccessible.
Not user friendly for millions.

The original pricing interface was a dense grid of numbers with no logical heading structure, no keyboard navigation and no screen reader compatibility. Lack of accessibility stood as a significant barrier to our legal and ethical goals.

Accessibility Failures

No ARIA roles or labels. Non-semantic table markup. Focus order not matching visual layout. Color-only indicators with no text alternative. No keyboard shortcut support.

Business Impact

Members abandoning pricing comparison flows. Increased call center volume. Section 508 compliance risk. High-friction checkout path affecting conversion.

Pricing Interface · Accessible card-based comparison layout · Express Scripts
Design Approach

Structure before style.

Before redesigning visuals, I rebuilt the semantic structure. Each pricing option became a card with a logical heading hierarchy, explicit text labels replacing color-only indicators, and a clear focus path for keyboard navigation.

Left: Pricing card with explicit text labels · Right: Healthcare context
Process

Testing-first methodology.

01
Accessibility Audit

Full WCAG 2.2 audit with manual screen reader testing (VoiceOver iOS/macOS, NVDA Windows). Every failure documented with severity and fix recommendation.

02
Semantic Redesign

Rebuilt IA with proper heading hierarchy. Replaced color-only indicators with text + icon pairs. Added ARIA roles, labels and live region announcements for dynamic price updates.

03
Visual Redesign

Clean card layout with 7:1 contrast on key elements. Large tap targets. Focus states at 3:1 minimum. Cognitive load reduction through whitespace and logical grouping.

04
Usability Testing

Sessions with visually impaired users using VoiceOver and NVDA. Iterated on announcement order and label wording based on direct feedback. Zero critical failures post-testing.

The Problem

An information dump
instead of a dashboard.

The member dashboard was a legacy list interface with no visual hierarchy, no status indicators and no motion feedback. Members couldn't quickly understand their prescription status, account activity or available actions. It displayed information. It didn't communicate it.

Dashboard card layout · NewRx motion · Status card detail
Design System

Cards, motion and meaning.

The redesign introduced a card grid where each card communicated a distinct status context. Micro-interactions and loading transitions made state changes visible and reassuring rather than abrupt.

Motion Philosophy

Every transition communicates a state change. Cards enter with staggered reveal. Loading uses skeleton screens not spinners. Status badges animate only on change, not on load.

Tools Used

Figma for layout and components. After Effects for motion specs. Framer for interactive prototyping. Storybook for component handoff and documentation.

The Concept

One metaphor.
Three states.

A fluid visual metaphor communicating the prescription process, from a traditional physical pill bottle, through a prescription label, to a clean digital circular element. The animation signals the shift from legacy pharmacy to digital-first experience. Every frame designed, animated and exported solo in After Effects.

NewRx · Pill bottle → label → circular digital ID · After Effects
Motion System

Purpose-first animation.

In healthcare, motion carries responsibility. It must communicate progress without inducing anxiety. Every easing curve was deliberate, ease-out on entry, ease-in on exit, spring curves on transformation states.

Technical Specs

Designed in After Effects. Frame-by-frame easing curves. Exported as optimised GIF and MP4. Animation budget under 400ms. CSS fallback for prefers-reduced-motion.

Accessibility

All animated elements have static fallback states. Motion is decorative, interface communicates fully without it. Respects prefers-reduced-motion at CSS and component level.

Case Study 05 - LEAF Design System
Cigna / Evernorth · Design Systems · 2020–2025

LEAF
Design System

60+
Components shipped
10+
Teams adopted
~40%
Faster handoff
The Problem

10 teams.
10 different UIs.

Evernorth's product portfolio had grown through acquisition and independently-operating teams. Visual inconsistency, duplicated design work, inaccessible components shipped repeatedly. No single source of truth. No shared compliance baseline.

Design Debt

Multiple unmaintained libraries. No token system. Hardcoded hex values. No documentation developers could self-serve from. Repeated accessibility rework every sprint.

Compliance Risk

Each team applied accessibility differently. WCAG 2.2 inconsistent. No shared testing protocol or accessible-by-default component baseline.

LEAF Design System · Token architecture · Cigna / Evernorth
Architecture

Tokens → Components
→ Documentation.

Three architectural layers. First: a design token system as a single truth shared between Figma and code. Second: 60+ components with every state and accessibility behavior documented. Third: Storybook and Confluence docs for developer self-service.

Token architecture · LEAF components applied in Benefit Overview
Process

Adopted, not mandated.

01
Audit & Inventory

Audited all 10 product teams. 200+ unique components collapsed into 60+ composable LEAF components. Naming conventions and taxonomy established.

02
Token Architecture

Primitive, semantic and component-level tokens. Figma variables → CSS custom properties pipeline. Storybook and Figma kept in sync.

03
Component Development

All states documented: default, hover, focus, active, disabled, error, loading. WCAG 2.2 built-in, not retrofitted.

04
Storybook & Docs

MDX stories, code examples and video tutorials. Teams onboard without designer involvement, self-serve compliance.

05
Governance & Mentorship

Contribution model, critique protocols, accessibility review checklists, versioning standards. Mentored junior designers through documentation practices.
